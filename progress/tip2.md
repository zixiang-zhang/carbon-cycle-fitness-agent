1. 🚀 第一阶段：冷启动与画像 (Onboarding)
场景： 用户第一次使用 App。

用户动作：

输入基础信息（年龄、性别、身高、体重）。

关键交互： 选择训练日程（例如：勾选“周二练腿”、“周四练背”）。这是碳循环的基础。

前端逻辑：

表单提交后，调用 POST /api/v1/profile。

Loading 状态： 显示“AI 正在计算你的 TDEE 和代谢类型...”。

后端 Agent 动作：

计算 BMR 和 TDEE。

根据训练日程，打上标签（周二=高消耗日，周四=中消耗日）。

2. 📅 第二阶段：策略展示 (Strategy Reveal)
场景： AI 生成了第一份计划。

用户动作：

查看生成的“周计划卡片”。

用户看到：“周二（高碳日）：碳水 300g” vs “周三（低碳日）：碳水 50g”。

点击“确认并开始”。

前端逻辑：

渲染一个日历组件或时间轴。

用不同颜色区分日子（如：🔴 高碳，🟢 低碳，🟡 中碳）。

3. 🏠 第三阶段：每日仪表盘 (The Daily Driver)
场景： 用户每天打开次数最多的页面。

核心展示：

Header： 显示“今天是：低碳日 (Low Carb)”。

Progress： 剩余额度（例如：还剩 20g 碳水，但还有 100g 蛋白没吃）。

Action： 一个显眼的“➕ 记录”按钮。

关键交互 (Chat/Log)：

用户点击“记录”，弹出对话框或相机。

用户发图：“这是我的午餐”。

Agent 回复 (流式)： “识别到牛肉汉堡（去酱）。热量 450kcal，碳水 30g。目前你的碳水额度只剩 10g 了，晚餐建议只吃蔬菜沙拉。”

UI 更新： 进度条自动发生变化。

4. 🔄 第四阶段：动态调整与复盘 (The Feedback Loop)
场景 A：短周期调整（日间）

触发： 用户在“低碳日”吃超标了。

交互：

前端收到后端返回的 warning 标记。

弹出 Toast 或 Modal：“检测到今日碳水严重超标。为了不影响减脂，Agent 建议将明天的‘中碳日’调整为‘低碳日’。是否执行？”

用户点击“执行”。

场景 B：长周期复盘（周间）

触发： 周一早上，App 提示“请输入今日空腹体重”。

交互：

用户输入体重。

Agent (Reflector) 生成周报：“本周体重下降 0.5kg，符合预期。下周保持当前宏量系数。”

或者：“体重未下降，分析原因为周末‘欺骗餐’热量过高。下周将削减 5% 的总热量。”

💡 给前端开发的建议
数据驱动视图 (State-Driven UI)： 你的前端页面应该完全由后端返回的 UserStatus JSON 驱动。

如果后端返回 current_day_type: "HIGH_CARB"，前端背景就变红，进度条变长。

如果后端返回 alert: "Plan Changed"，前端就弹窗。

对话框不仅仅是聊天： 不要把 Chat 界面只当成文字聊天。Agent 返回的消息里可以包含 UI 组件指令。

后端返回： {"type": "food_card", "data": {"name": "Apple", "calories": 50}}

前端渲染： 一个漂亮的食物卡片，而不是一行干巴巴的文字。

优先开发顺序：

Onboarding (画像) + Dashboard (看数据) —— 先让数据能显示出来。

Chat/Vision (记录) —— 能够通过 API 更新数据。

Planning (生成) —— 把生成逻辑接通。
