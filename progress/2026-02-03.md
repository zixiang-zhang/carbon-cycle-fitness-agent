# 2026-02-03 开发进度总结

## 1. 后端核心架构修复与升级 (Backend Core)

### 1.1 数据库持久化层 (Repository & Storage)
针对“前端调整计划无法保存”的问题，对后端存储逻辑进行了深度重构：
*   **`app/db/repositories/plan_repo.py`**:
    *   新增 `update` 方法：实现了复杂的嵌套对象更新逻辑。
    *   **深度更新策略**：由于 `CarbonCyclePlan` 包含多个 `DayPlan` 子对象，直接更新会有脏数据风险。实现的逻辑是：先通过 `selectinload` 加载计划，清空旧的 `DayPlan` 记录，再根据前端提交的最新数据重新批量插入 `DayPlanModel`，确保数据绝对一致。
    *   **类型安全**：在模型转换层（Pydantic <-> SQLAlchemy）增加了更严格的类型检查。
*   **`app/db/db_storage.py`**:
    *   暴露了 `update_plan` 接口，打通了上层服务与底层 Repo 的调用链路。

### 1.2 API 接口层
*   **`app/api/plan.py`**:
    *   升级 `PATCH /plans/{id}` 接口：从简单的字段更新升级为支持嵌套模型更新。
    *   修正 `create_plan` 接口：将其改为 **异步 (async)** 调用，以适配新引入的 AI 增强服务。

## 2. 新功能：AI 智能计划增强 (Agent-Powered Plan Enrichment)

这是今日最大的架构突破，将系统从“死板公式”升级为“AI 智能生成”。

### 2.1 新增 AI 服务 (`app/services/plan_enrichment.py`)
创建了独立的 `PlanEnrichmentService`，其核心工作流如下：
1.  **上下文构建**：接收基础计算生成的 `DayPlan`（包含冷冰冰的热量/碳水数值）和 `UserProfile`（用户画像）。
2.  **Prompt 工程**：动态构建 Prompt，将用户的目标（如“减脂”）、当日类型（如“高碳日”）和宏量营养素目标填入模板。
3.  **LLM 调用**：
    *   调用 `app/llm/client.py` 中的 `LLMClient`。
    *   连接 **Aliyun Bailian (Qwen-Plus)** 模型。
    *   强制模型输出 **JSON 格式**，包含 `training` (训练摘要) 和 `diet` (食谱建议) 两个字段。
4.  **容错处理**：实现了 Fallback 机制。如果 LLM 由于网络原因超时或报错，服务会自动回退到一套高质量的静态文案，确保用户体验不受影响。

### 2.2 策略服务重构 (`app/services/carbon_strategy.py`)
*   **流程改造**：将 `generate_plan` 方法彻底异步化。
*   **Pipeline 设计**：
    *   Step 1 (算法): 基于 TDEE 公式计算每日热量缺口。
    *   Step 2 (算法): 基于碳循环规则（4天循环/7天循环）分配每日碳水比例。
    *   Step 3 (AI): **(新增)** 调用 `PlanEnrichmentService`，为生成的每一天“注入灵魂”，生成具体的动作指导和菜单。

## 3. 前端交互优化 (Frontend)

### 3.1 注册流程 (Onboarding)
